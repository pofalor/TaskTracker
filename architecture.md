# Архитектура системы управления задачами и рабочим временем

## Общее описание
Система предназначена для управления задачами и рабочим временем сотрудников компаний.

## Рабочее пространство (Workspace)

### Типы рабочих пространств
- **Личное рабочее пространство**:
  - У каждого пользователя может быть только **одно** личное рабочее пространство
  
- **Корпоративное рабочее пространство**:
  - Должно иметь **уникальное имя** в рамках одного управляющего (создателя)
  - Запрещены дубли:
    - ИНН организации
    - Комбинация: **дата регистрации + юридический адрес + страна**

## Информация о рабочем пространстве (Projects)

### Основные разделы
1. **Проекты**:
   - Список проектов в выбранном рабочем пространстве
   
2. **Приглашения**:
   - Управляющий может:
     - Приглашать пользователей по **email** или **никнейму** (поиск только по полному совпадению)
     - Удалять пользователей из организации
   - Пользователи:
     - Видят приглашения (как отправленные им, так и полученные)
     - Могут принимать или отклонять их
     - Не могут получать новые приглашения, пока не обработают текущее
   - Ограничения:
     - Приглашения **нельзя изменить или удалить** после отправки
     - Видны только **ожидающие приглашения**

## Техническая реализация

### Сущности и процессы
- **`UserWorkspaceStatusChange`**:
  - Создается при отправке приглашения
  - При принятии (`IsConfirmed = true`):
    - Создается `WorkspaceMember`
  - При отклонении:
    - `WorkspaceMember` не создается
  - При удалении пользователя:
    - Создается новая `UserWorkspaceStatusChange`
    - `WorkspaceMember` помечается статусом `"Deleted"`

- **Создание рабочего пространства**:
  - Автоматически создается `WorkspaceMember` для создателя с ролью `Owner`

## Новый функционал

### 1. Автоматическое списывание часов
- Добавляется **кнопка** для автоматического трекинга времени

### 2. Администрирование компаний
- **Администратор** проверяет и подтверждает новые компании:
  - **Личные** рабочие пространства не требуют подтверждения
  - **Корпоративные** требуют обязательного подтверждения
- **Правила подтверждения**:
  - Администратор не может подтвердить собственное рабочее пространство. Подтверждение должно выполняться другим администратором
- **Правила отображения**:
  - Создатель видит статус (`Pending`, `Approved`, `Rejected`)
  - Остальные пользователи **не видят** пространство до подтверждения
  - Неподтвержденные пространства видны только:
    - Создателю пространства
    - Администраторам системы

### 3. Отчёты
- Функционал в разработке (не срочно)

### 4. Удаление из рабочего пространства
- При удалении пользователя **автоматически останавливается** трекинг его задач
